## 3. 지금 수정해야 할 포인트 (우선순위순)

### P1-1. 클로가 여전히 안 보임 (플레이 불가능 수준) ✅→수정 필요
- 근거: 스크린샷에서 확인. home y=2.2로 내렸어도 **이동 중(y=topY=2.2) 클로가 마퀴 스트립(y 2.06~2.34)과 하우징에 가려짐**. 조준하는 내내 클로가 안 보임
- 추가 아티팩트: 레일(y=2.95)·트롤리(y=2.93)가 **하우징 지붕(상단 2.8) 위에 떠 있음** — 지붕 위에 트롤리가 얹혀 보임
- 수정안: `CLAW.topY/home.y = 1.9`, 레일/트롤리 y를 유리 내부(≈2.05/2.03)로, `RAIL_Y/TROLLEY_Y`를 config로 이동 (규칙 3)

### P1-2. `setMode`에 IDLE 가드 없음 (크래시 위험)
- 지시서 §5: "토글은 clawPhase==='IDLE'일 때만" — 현재 store.setMode는 무조건 전환
- 위험 시나리오: 운반 중(조인트 부착) 운영자 모드 진입 → "비우기"로 인형 제거 → **살아있는 조인트가 참조하는 바디 제거 → rapier 크래시** 가능
- 수정안: `setMode`에서 `get().clawPhase !== 'IDLE'`이면 무시 (store 레벨 방어가 UI 가드보다 안전)

### P2-1. 창 포커스 아웃 시 키가 눌린 채 고착
- 키 홀드 중 Alt-Tab하면 keyup을 못 받아 `controlsState`가 true로 남음 → 복귀 시 클로가 혼자 이동
- 수정안: `useControls`에 `window 'blur'` 리스너 추가 → btnX/btnZ 리셋 (3줄)

### P2-2. 승리 판정 타이밍 경계 (Phase 4 설계 주의)
- 인형이 느리게 굴러 `ROUND_END` 이후 센서 진입 가능 → 점수는 반영되나 결과 토스트가 "실패"로 뜰 수 있음
- 수정안: Phase 4에서 토스트를 ROUND_END 순간이 아니라 **RESULT 표시 중 `wonThisRound`를 구독**해 갱신

### P3 (기록만, 지금 수정 불요)
- **키 충돌**: Phase 4 디버그 슬라이더에 포커스 시 ArrowRight/Up이 슬라이더와 클로를 동시 조작 → input/slider 포커스 시 키 무시 처리 필요
- **번들 3.3MB 경고**: rapier WASM base64 내장 때문(예상됨). 로컬/교육용엔 무해. 배포 시 `chunkSizeWarningLimit` 조정 또는 manualChunks
- **dev 전용 전역** `__clawStore`/`__clawControls`: `import.meta.env.DEV` 가드됨 — 프로덕션 제외 확인됨
- **useFrame 이벤트 처리 위치**: GRAB_NOW 처리에서 `Math.random` 사용은 런타임 한정(순수 로직은 RNG 주입) — 규칙 위반 아님

## 4. 남은 [C] 검증 체크리스트 (탭을 화면에 띄우고 확인)

1. ①(→/Z) 홀드 → X 이동, ②(↑/X) 홀드 → Z 이동, 릴리스 → 자동 하강 (R2.1~2.3)
2. 하강 → 손가락 닫힘 → 인형 매달려 운반(달랑거림) → 배출구 투하 → 크레딧/점수 변화 (R3.3, R3.5)

### 결론
계획 대비 구조적 이탈 없음. 남은 리스크는 전부 "기록만" 수준(초기 낙하 중 우연한 승리, 3.3MB 번들, 모바일 실기기 미확인) — 기능적으로 막는 것은 없음.

### 신규 관찰 — 우선순위 낮음, 기록만
- **초기 낙하 중 자연 발생 승리**: 인형 더미가 정착하는 과정에서 인형 1개가 우연히 배출구 샤프트로 굴러 들어가 크레딧 소모 없이 승리 판정되는 현상을 재현 확인(로드마다 발생하지는 않음). `ChuteSensor`가 클로 상태와 무관하게 항상 감시하는 의도된 설계(R3.5, 실제 기계와 동일 원리)의 자연스러운 결과이며 버그는 아니지만, 사용자에게는 "이유 없이 승리 토스트가 뜨는" 것처럼 보일 수 있음. Phase 6에서 원치 않으면 초기 스폰 시 배출구 샤프트 진입 마진을 좀 더 넓히는 정도로 완화 가능(선택 사항)
- **테스트 방법론 노트**: 브라우저 자동화로 Phase 5를 검증하던 중, 파일을 연속 수정하며 Vite HMR이 적용되는 시점에 클릭하면 컴포넌트가 일시적으로 깨진 상태(예: 방금 추가한 상수가 아직 모듈에 반영 안 됨)에서 클릭이 무시되는 현상 발견 — **실제 앱 버그 아님**, 편집 중 라이브 리로드와 자동화 클릭이 겹친 테스트 아티팩트. 안정적으로 재현하려면 편집을 마친 뒤 페이지를 새로고침(hard reload)하고 검증할 것
