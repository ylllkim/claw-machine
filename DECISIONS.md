# 인형뽑기 프로젝트 — 결정 히스토리 (왜 이렇게 정했는가)

`PLAN.md`(구현 지시서)·`SPEC.md`(수용 기준)는 "무엇을/어떻게"를 담고 있고, 이 문서는 각 결정의 "왜"를 압축해서 담는다. Claude Desktop 등 다른 도구로 이어갈 때 이 파일을 먼저 읽으면 맥락 손실 없이 합류할 수 있다.

## 프로젝트 목적·제약

- 사내 교육용 **재미 프로젝트**. 프레임워크/엔진 학습이 목적이 아니라 **결과물이 가장 인형뽑기 같아 보이는 것**이 목표
- 바이브 코딩으로 진행, 이후 Vercel 호스팅 예정이나 **1차는 로컬 실행만** 확인
- **라이선스 리스크 제로**가 하드 제약: 모든 의존성·에셋은 MIT/Apache-2.0/CC0만 허용
- 물리엔진은 `npm install`만으로 동작해야 함 — 네이티브 빌드, 에디터 툴링, 유료 티어 전부 배제

## 스택 결정과 비교 근거

### 프레임워크: React Three Fiber (확정)
| | R3F (채택) | Vanilla Three.js | Babylon.js |
|---|---|---|---|
| 근거 | 선언적 JSX라 바이브 코딩 반복이 작은 diff로 끝남. pmndrs 생태계 활발 | 물리↔메시 동기화 보일러플레이트가 매 반복마다 늘어남 — 교육 목적이 "Three.js 자체 학습"이 아니라면 손해 | **탈락**: 대표 물리엔진 Havok이 프로프라이어터리(라이선스 제약 위반). 대안(cannon-es/ammo.js)은 아래 표에서 보듯 더 나쁨 |
사용자 확인: "1회성 재미 프로젝트, 프레임워크 학습과 무관, 가장 인형뽑기 같은 동작이 목표" → 결과물 속도/품질 최우선인 R3F로 확정.

### 물리엔진: Rapier (`@dimforge/rapier3d-compat`, Apache-2.0)
| | **Rapier (채택)** | cannon-es | ammo.js |
|---|---|---|---|
| 라이선스 | Apache-2.0 | MIT | zlib |
| 유지보수 | 활발 (2025/26 기준 정기 릴리스) | **사실상 중단** (12개월+ 릴리스 없음) | 휴면, DX 최악 |
| 관절/모터 (집게 필수) | 최상 — 런타임 조인트 생성/제거 API 깔끔 | 취약, 드리프트 심함 | 강력하나 C++ 이식 흔적으로 사용 난해 |
| 쌓기 안정성 (인형 더미) | 안정적 슬리핑 | 지터로 악명 | 양호하나 튜닝 어려움 |
| 설치 편의성 | `-compat` 패키지가 WASM을 base64로 내장 → 번들러 설정 제로 | 순수 JS, 설치는 쉬움 | WASM 초기화 번거로움 |

집게 메커니즘이 런타임 조인트 생성/제거에 크게 의존하므로, 이 부분이 가장 약한 cannon-es는 초반에 배제. Havok(Babylon 기본)은 라이선스 문제로 원천 배제.

### 잡기 메커니즘: 하이브리드 (손가락은 비주얼, 판정은 센서+확률, 부착은 조인트)
| | (a) 완전 물리 그립 | (b) 순수 페이크 그랩 | **(c) 하이브리드 (채택)** |
|---|---|---|---|
| 방식 | 키네마틱 손가락+마찰로 실제 파지 | 센서 감지+확률+즉시 부착, 손가락 비주얼과 무관 | 손가락은 애니메이션(콜라이더 없음), 판정은 반경 내 최근접+확률, 부착은 스페리컬 조인트 |
| 문제 | 튜닝 지옥 — 물체가 튀거나 지터 발생, 소규모 프로젝트엔 부적합 | 손가락이 인형을 시각적으로 관통하면 "가짜"티가 남 | (a)의 체감을 (b)의 안정성으로 확보. 조인트 덕에 인형이 달랑거리며 흔들림(리얼함 공짜로 얻음). **실제 인형뽑기 기계도 그립 강도를 확률로 조작함** — 교육 포인트로도 좋음 |

이 결정 때문에 v1 클로에는 **콜라이더가 없음**(의도적) — 물리 간섭으로 인한 예측불가 동작을 원천 차단.

## 조작 방식: 정통 2버튼 (확정)
버튼① 홀드=X 이동, 버튼② 홀드=Z 이동, ② 릴리스 시 자동 하강 → 자동 그랩 → 자동 복귀 → 자동 릴리스. 실제 오락실 인형뽑기 기계와 동일한 방식. (대안이었던 "자유 이동+드롭 버튼"보다 더 인형뽑기답다는 이유로 선택.)

## 승률 리깅: 확률 + 디버그 패널 (확정)
`gripStrength`를 0~1 확률로 노출하고 화면에 슬라이더로 표시. 실제 클로 머신 운영 원리(그립 강도를 조작해 배당률 관리)를 그대로 재현 + 교육 시연 포인트로 활용. 순수 함수(`rollGrab`, RNG 주입식)라 고정 시드로 단위 테스트 가능.

## 인형 다양성 + 운영자 모드 (v2에서 추가된 요구사항)
최초 계획(v1)은 단일 종류 절차적 인형이었으나, 사용자가 다음을 요청해 v2로 확장:
1. **여러 종류의 인형** — 곰/토끼/고양이/펭귄/병아리 5종, 절차적 프리미티브 조합(구+캡슐)으로 제작해 라이선스 리스크 없이 다양성 확보. 카탈로그 데이터 1건 추가만으로 종류 확장 가능하게 설계(`src/catalog.ts`)
2. **운영자 모드** — 인형뽑기방 운영자처럼 인형 종류를 고르고 클릭으로 배치/삭제, 레이아웃을 저장·불러오기. "너무 어려울까?" 라는 질문에 대한 판단: 난이도 낮음~중간으로 충분히 가능 — R3F의 클릭 이벤트(`onClick`)와 raycast 평면만으로 구현되고, 별도 드래그 이동 같은 고난도 기능은 v1 범위에서 제외했기 때문

## 진행 방법론: SDD + 부분 TDD (v2에서 추가)
사용자가 "바이브 코딩 결과가 체계적으로 나오도록" 요청 → 다음 하이브리드로 확정:
- **SDD(Spec-Driven)**: `SPEC.md`에 요구사항 ID(R1~R6)와 수용 기준을 먼저 못박고, Phase 완료 조건으로 사용
- **TDD는 순수 로직에만 적용**: `src/logic/`(상태전이·확률판정·레이아웃 직렬화)은 Three/Rapier 의존이 없어 vitest로 선-테스트 가능. 물리 시뮬레이션·렌더링·입력 체감은 3D 특성상 단위 테스트 비용 대비 가치가 낮다고 판단해 **브라우저 플레이스루 체크리스트**로 대체 (SPEC.md의 `[T]`/`[C]` 태그로 구분)

## 모델 전환 관련 맥락 (참고용, 기술 결정 아님)
설계·계획 수립은 Fable 5(reasoning effort xhigh)로 진행했고, 사용자가 이후 실제 코드 구현은 Sonnet으로 전환하기로 결정. 이 때문에 `PLAN.md`는 "Sonnet이 그대로 따라 구현할 수 있는 지시서" 수준까지 구체화됨(좌표값, API 시그니처 확인 결과, 파일별 정확한 작업 항목 포함) — 설계 판단을 실행 시점에 다시 하지 않아도 되도록 하기 위함.
